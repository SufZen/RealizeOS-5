# Intent 1.1 — Security Audit for Public Repository

## Goal

Scan and clean the entire RealizeOS V5 codebase so it is safe to make the repository public as part of the BSL open-source launch.

## Context

RealizeOS is transitioning from a private closed-source product to a Business Source Licensed open-source project. Before the repo goes public, every hardcoded secret, API key, internal URL, and credential must be found and removed. A half-baked first impression is nearly impossible to recover — a leaked secret on day 1 would undermine trust in the entire project.

## Scope

### IN (must do)

- Scan all Python, YAML, JSON, `.env`, and config files for hardcoded secrets, API keys, internal URLs, and credentials
- Remove or parameterize all findings (replace with environment variable references)
- Add a `.gitignore` entry for all sensitive file patterns (`.env`, `*.key`, `*.pem`, credential files)
- Add `detect-secrets` baseline file and integrate into CI pipeline
- Verify no Google Workspace / Stripe / LLM provider keys leak in example configs or test fixtures
- Scan git history for previously committed secrets (recommend BFG or git-filter-repo if found)
- Review and sanitize all `.env.example` or sample config files

### OUT (explicitly excluded)

- Dependency vulnerability scanning (separate future intent)
- Architecture restructuring
- Code refactoring unrelated to secret removal
- Dashboard code security review (separate intent)

## Acceptance Criteria

- [ ] Zero hardcoded secrets, API keys, or internal URLs remain in the codebase
- [ ] `detect-secrets` baseline file (`.secrets.baseline`) exists and scan passes cleanly
- [ ] `.gitignore` covers `.env`, `*.key`, `*.pem`, `realize_data.db`, and other sensitive patterns
- [ ] CI pipeline (`.github/workflows/`) includes a secret scanning step
- [ ] All parameterized values use environment variable references with sensible defaults documented
- [ ] No breaking changes to existing functionality (tests still pass)

## Constraints

- Must not break existing API contracts or configuration loading
- Must not remove any functionality — only parameterize hardcoded values
- Must maintain backward compatibility with existing `.env` configurations
- All changes must follow existing patterns in `project-context.md`

## Build Mode

**Mode:** `standard`

## Additional Notes

- This is the first intent in a 16-intent implementation plan. See `.autobuild/` for the full plan context.
- The launch strategy specifies BSL 1.1 licensing with a 4-year change date to Apache 2.0.
- Priority scanning paths: `realize_core/`, `realize_api/`, `config/`, `ventures/`, `tests/fixtures/`
- Use `detect-secrets` (Yelp) or `gitleaks` for automated scanning.
