⚙️ Processing Engine

The processing engine is the core of RealizeOS — it receives messages from channels, routes them to the right agent, and orchestrates the full response pipeline.

Message Pipeline

Every message passes through this pipeline in base_handler.py:

1
Session Check
Is there an active creative session? If yes, continue it.
▼
2
Skill Detection
Does the message match a skill trigger? If yes, execute the skill workflow.
▼
3
Agent Routing
Score agents by keyword match → select the best agent for this message.
▼
4
Task Classification
Classify as simple / content / complex → select LLM model tier.
▼
5
Prompt Assembly
Build the 12-layer prompt from knowledge base, agent definition, memory, and context.
▼
6
LLM Call + Tools
Send to selected model. Execute tool calls if needed. Return response.

Configuration

RealizeOS is configured through realize-os.yaml and .env:

# realize-os.yaml
name: "My Business"

systems:
  - key: my-business
    name: "My Business"
    directory: systems/my-business

    routing:
      content: [writer, reviewer]
      strategy: [analyst, orchestrator]
      general: [orchestrator]

    agent_routing:
      writer: [write, draft, post, blog, content]
      analyst: [analyze, research, data, market]
      reviewer: [review, check, quality, approve]
      orchestrator: [plan, help, think, prioritize]

features:
  review_pipeline: true      # Auto-route to reviewer
  auto_memory: true          # Log learnings
  proactive_mode: true       # Proactive suggestions
  activity_log: true         # SQLite activity logging
  approval_gates: false      # Human-in-the-loop

llm:
  default_model: claude-sonnet
  routing:
    simple: gemini-flash
    content: claude-sonnet
    complex: claude-opus

Creative Sessions

When a task requires multiple steps (write → review → iterate), RealizeOS creates a creative session:

  • Stage progression: briefing → drafting → iterating → reviewing → approved
  • Pipeline tracking: Which agents are involved and in what order
  • Version history: All drafts and revisions are tracked
  • Context persistence: Session state persists across messages
# Check active session
GET /api/systems/{system_key}/sessions/{user_id}

🧠 Intelligence Layer

The intelligence layer is where RealizeOS transforms a raw message into a thoughtful, context-rich response using agents, skills, LLM routing, and multi-layer prompt assembly.

Agents

Agents are specialized AI team members, each with a unique role, personality, and capabilities. Agents are defined as markdown or YAML files in A-agents/.

Built-in Agents

🎯

Orchestrator

General coordinator, task router, and planner. Handles requests that don't match specialized agents.

✍️

Writer

Content creation specialist — blog posts, emails, social media, newsletters.

🔍

Reviewer

Quality gatekeeper with scoring framework. Reviews drafts against venture standards.

📊

Analyst

Research, strategy, data analysis. Competitive analysis, market trends, decision support.

V2 Agent Definition

# A-agents/writer.yaml
version: "2"
name: writer
scope: Content creation specialist
communication_style: friendly, professional
inputs:
  - content_brief
  - venture_voice
outputs:
  - draft_content
  - revision_notes
guardrails:
  require_approval_for:
    - publish
    - send_email
tools:
  - gmail_send
  - drive_create_doc
critical_rules:
  - Always match the venture voice
  - Include a call-to-action
success_metrics:
  - content_quality_score > 80
ℹ️ V1 agent definitions still work. The loader auto-detects the format — no migration required.

Skills & Workflows

Skills are YAML-defined workflows in R-routines/skills/ that chain multiple steps.

v1 Skills — Simple Pipelines

name: content_pipeline
triggers:
  - "write a post"
  - "create content"
task_type: content
pipeline:
  - writer
  - reviewer

v2 Skills — Multi-Step Workflows

name: competitor_analysis
version: "2.0"
triggers:
  - "analyze competitor"
  - "competitive analysis"
steps:
  - id: search
    type: tool
    action: web_search
    params:
      query: "{user_message} site:linkedin.com"

  - id: analyze
    type: agent
    agent: analyst
    inject_context: [search]
    prompt: |
      Analyze the competitive landscape for: {user_message}

  - id: confirm
    type: human
    question: "Create a detailed report?"

  - id: report
    type: agent
    agent: writer
    inject_context: [search, analyze]

conditionBranch logiccheck, branches

Knowledge Base & Search

RealizeOS uses hybrid KB search to find relevant context for each message:

  • FTS5 — SQLite full-text search for keyword matching
  • Vector embeddings — Semantic similarity search for concept matching
  • Hybrid scoring — Combines both scores for optimal retrieval

Hooks & Cron

Event Hooks (Pub/Sub)

Subscribe to lifecycle events across the system:

  • on_message — Every incoming message
  • on_venture_change — Venture config changes
  • on_agent_complete — Agent finishes processing
  • on_skill_trigger — Skill is activated

    config:
      jobs:
        - id: daily_digest
          trigger: cron
          hour: 9
          action: run_skill
          params:
            skill: daily_summary

🏗️ Infrastructure

The infrastructure layer provides pluggable storage, knowledge base search, self-evolution, and optimization capabilities.

Storage System

RealizeOS uses a pluggable storage architecture with multiple backend support:

Storage Backends

💾

Local Storage

Filesystem-based storage with atomic writes, directory sync, and automatic cleanup. Default backend.

☁️

S3 Storage

S3-compatible object storage (AWS S3, MinIO, DigitalOcean Spaces). Supports multipart uploads and presigned URLs.

Sync Manager

The sync manager coordinates background synchronization between storage backends:

  • Bi-directional sync — Push local changes to S3, pull remote changes
  • Conflict resolution — Last-write-wins with configurable strategies
  • Orphan cleanup — Detect and remove orphaned files
  • Event logging — All sync operations logged to storage_sync_log

Self-Evolution Engine

RealizeOS learns and improves over time with user approval:

1
Interaction Tracking
Every request is logged with metadata — agent, model, tokens, success
▼
2
Gap Detection
Find unhandled patterns, repeated ad-hoc requests, low-quality responses
▼
3
Skill Suggestion
Auto-generate skill YAML for detected gaps
▼
4
Prompt Refinement
Suggest agent prompt improvements from feedback
⚠️ All evolution changes require user approval before being applied.

Optimization Engine

Run A/B experiments on prompts, models, and parameters to improve performance:

  • Experiment Engine — Define experiments with control/variant groups
  • Experiment Tracker — Track results and statistical significance
  • Metrics — Define custom metrics (quality, latency, cost, tokens)
  • Token Optimization — Compress prompts to reduce token usage without losing quality

🔒 Security

RealizeOS V5 includes a comprehensive security layer with authentication, authorization, threat detection, and audit logging.

JWT Authentication

Secure API access with JSON Web Tokens:

  • Token issuance — Sign tokens with configurable expiration
  • Token validation — Verify signatures and check expiration
  • Claims — User ID, roles, permissions encoded in token payload
  • Refresh flow — Refresh tokens for long-lived sessions
# Environment configuration
JWT_SECRET_KEY=your-secret-key-here
JWT_ALGORITHM=HS256
JWT_EXPIRATION_MINUTES=60

Role-Based Access Control (RBAC)

Define roles and permissions in YAML with fine-grained access control:

# security/roles.yaml
roles:
  admin:
    permissions:
      - system:manage
      - venture:create
      - venture:delete
      - agent:configure
      - security:audit
  operator:
    permissions:
      - venture:read
      - venture:write
      - agent:use
  viewer:
    permissions:
      - venture:read
      - agent:use

The RBAC manager loads roles from YAML files and validates permissions against JWT token claims.

Prompt Injection Defense

Multi-layer defense against prompt injection attacks:

🛡️

Pattern Detection

Regex-based scanning for known injection patterns (ignore instructions, role override, etc.)

🧹

Input Sanitization

Strip dangerous characters and normalize input before processing

📏

Threat Scoring

Calculate risk score from 0-1 based on multiple signals

🚫

Auto-Block

Reject requests exceeding configurable threat threshold

Audit Logging

Every security-relevant action is logged with full context:

  • Authentication events — Login, logout, token refresh, failures
  • Authorization checks — Permission grants and denials
  • Security threats — Injection attempts, suspicious patterns
  • Admin actions — Config changes, user management
💡 Audit logs are stored in SQLite and can be queried via the API or exported for compliance.

Activity Logging

When activity_log: true is set, every system event is logged to SQLite:

  • Message received / response sent
  • Agent selected / skill triggered
  • Tool executed / LLM called
  • Error / warning events

The dashboard displays these in a real-time SSE-powered activity feed.

Approval Gates

When approval_gates: true is set, consequential actions require human approval before execution:

  • Sending emails
  • Creating calendar events
  • Publishing content
  • Any action marked in agent guardrails

    listen 443 ssl http2;
    server_name realize.yourdomain.com;

    ssl_certificate /path/to/fullchain.pem;
    ssl_certificate_key /path/to/privkey.pem;

        proxy_pass http://127.0.0.1:8080;

        proxy_buffering off;        # SSE support

}

⭐ Best Practices

Venture Setup

  • Start with a template closest to your business type
  • Fill in F-foundations/identity.md thoroughly — this drives all agent responses
  • Write clear voice rules in F-foundations/voice-guide.md
  • Add real domain knowledge into B-brain/ — the more context, the better the outputs

Agent Design

  • Keep agents focused — one primary responsibility per agent
  • Use V2 format for agents that need guardrails or tool access
  • Define clear critical_rules to prevent common mistakes
  • Add 5-8 routing keywords per agent to avoid misrouting

Skill Authoring

  • Use 3-5 specific triggers per skill to avoid false matches
  • Add human steps before any destructive actions
  • Use inject_context to chain step results
  • Test skills with varied phrasings of the trigger

Security Hardening

  • Always set REALIZE_API_KEY in production
  • Enable JWT authentication for multi-user deployments
  • Use RBAC to restrict access by role
  • Enable audit logging for compliance
  • Set rate limits to prevent abuse
